# AP-001: Capability-Driven Architecture

> **状态**: `proposal`
> **作者**: Hermes Architecture Team
> **创建日期**: 2026-07-03
> **升级条件**: Observation Week 结束 + Environment Registry PoC 完成 + 至少 2 个 Capability 接入成功

---

## 1. Motivation

Hermes 当前以 Tool 为中心，每个软件需要单独接入和维护。Proposal 希望演进到 **Capability 为中心**，降低软件耦合，提高环境适配能力。

> **Hermes 不应该学习"如何使用 VS Code"，而应该学习"如何操作 Workspace"；软件只是 Workspace Capability 的一种 Adapter。**

---

## 2. Goals

- Tool 解耦 — Router 不直接感知具体软件
- Adapter 可替换 — 同一种 Capability 可有多种实现
- Environment 感知 — Hermes 知道电脑上有什么 / 什么在运行
- 按需加载 Capability — 不加载不必要的上下文

---

## 3. Architecture

```
Task
  │
  ▼
Capability (Workspace / Repository / Search / Browser / Runtime)
  │
  ▼
Adapter (VS Code / Git / Chrome / Docker)
  │
  ▼
Software (具体安装)
```

Router 永远只问：**"我需要什么 Capability？"**
Adapter 回答：**"我有哪种实现？"**
Software 回答：**"我是否可用？"**

---

## 4. Components

### 4.1 Capability Registry

Capability → Adapter 映射关系。

```yaml
workspace:
  adapters: [vscode, idea]
  default: vscode
repository:
  adapters: [git, gh]
  default: git
```

### 4.2 Adapter

同一 Capability 的不同实现，可替换。

| Capability | Adapter | 软件 |
|-----------|---------|------|
| Workspace | vscode | VS Code |
| Workspace | idea | IntelliJ IDEA |
| Repository | git | Git CLI |
| Search | everything | Everything (Windows) |
| Search | ripgrep | ripgrep (跨平台) |

### 4.3 Environment Registry

电脑真实状态快照，Router 查询使用。

```yaml
vscode:
  installed: true
  running: true
  workspace: "C:/Users/32295/IdeaProjects/ExamBed"
docker:
  installed: true
  running: false
ollama:
  online: true
everything:
  installed: false
```

## 5. Capability Priority

| 优先级 | Capability | First Adapter | 原因 |
|-------|-----------|--------------|------|
| 🥇 P0 | Workspace | VS Code | 当前开发上下文 |
| 🥇 P0 | Repository | Git + gh CLI | 代码变更和历史 |
| 🥇 P0 | Search | Everything（安装后） | 毫秒级文件定位 |
| 🥈 P1 | Browser | Chrome | 标签页/书签上下文 |
| 🥈 P1 | Runtime | Docker / WSL | 运行环境感知 |

---

## 6. Phase Plan

### Phase 2a（Environment Registry）

仅建立 Environment Registry，只读验证层。

**产出**: `environment/registry.json`，Hermes 启动时自动检测并写入。

**Router 行为**: 开始查询环境状态，但不改变决策逻辑。

**允许在 Feature Freeze 内做**: ✅ 因为只读，不改变运行时行为。

### Phase 2b（Capability Registry）

建立 Capability → Adapter 映射，Router 开始按 Capability 路由。

**产出**: `capabilities/*.yaml`

### Phase 2c（Capability Library）

按需加载 Capability，Tool Gating 联动。

**产出**: 动态 Capability 发现和加载机制。

## 附录 A: 设计原则

1. **Capability First, Adapter Second, Software Last.**
   — 用户说需求，Hermes 想 Capability，Router 选 Adapter。

2. **每增加一层抽象，都必须能消除至少两个具体问题；否则只是新的复杂度。**

---

## 附录 B: 风险矩阵（L0-L3 替代方案）

Kimi Code 指出当前 L0-L3 混了"操作性质"和"恢复难度"两个维度。
建议后续改为二维矩阵：

| | 本地影响 | 外部影响 |
|---|---------|---------|
| **可逆** | L0: 创建文件 | L1: 创建 Git Branch |
| **不可逆** | L2: 删除本地文件 | L3: Git Push / 删除远程资源 |

L0-L3 保留作为权限等级名称，底层逻辑改为二维判定。
